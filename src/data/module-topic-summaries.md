# Module Topic Summaries

## pqc-101 — PQC 101

**Scope.** PQC 101 is the foundational introduction to post-quantum cryptography, covering why current public-key algorithms (RSA, ECC, ECDSA, EdDSA, ECDH, DH) are vulnerable to quantum computers via Shor's algorithm, the three families of NIST-standardized PQC algorithms (lattice-based ML-KEM/ML-DSA, hash-based SLH-DSA/LMS/XMSS, and code-based schemes), and the HNDL (Harvest Now, Decrypt Later) threat model that makes quantum migration urgent regardless of CRQC timeline certainty. The module walks through the 2016–2035 migration timeline from NIST standardization through NSA CNSA 2.0 compliance deadlines, industry-specific urgency levels, and role-based next steps. Out of scope: deep cryptanalysis, detailed algorithm implementations, quantum physics mechanics, and post-quantum signature variants beyond the NIST-selected families.

**Sub-topics keywords:** Shor's algorithm, Grover's algorithm, quantum threat, lattice-based cryptography, hash-based signatures, code-based schemes, NIST FIPS 203 204 205, CNSA 2.0, HNDL attack, HNFL attack, algorithm families, key size comparison, CRQC timeline, quantum threat timeline, classical vs post-quantum.

## quantum-threats — Quantum Threats

**Scope.** This module provides in-depth technical explanation of how quantum computers break cryptography: qubits and superposition, Shor's algorithm solving integer factorization (RSA) and discrete logarithm (ECC/DH) in polynomial time O(n³), Grover's algorithm providing quadratic speedup for symmetric weakening, and CRQC timeline projections from NIST, NSA, Global Risk Institute, BSI, and ANSSI. Covers the HNDL and HNFL attack phases, security level degradation calculator, algorithm vulnerability matrix, and Mosca's Theorem for calculating migration deadlines. Emphasizes quantitative analysis: RSA-2048 requires ~4,098 logical qubits; P-256 ~2,330 qubits; AES-256 remains 128-bit secure under Grover's. Out of scope: quantum physics deeper than superposition/entanglement basics, quantum error correction details, CRQC engineering challenges, and quantum algorithms beyond Shor's and Grover's.

**Sub-topics keywords:** Shor's algorithm, Grover's algorithm, qubits, superposition, entanglement, CRQC cryptographically relevant quantum computer, HNDL harvest now decrypt later, HNFL harvest now forge later, security level degradation, key size, ciphertext size, logical qubit count, RSA vulnerability, ECC vulnerability, AES-256 quantum safety, Mosca's theorem, CRQC timeline, NIST IR 8547.

## hybrid-crypto — Hybrid Cryptography

**Scope.** Hybrid cryptography combines classical and PQC algorithms for defense-in-depth during the quantum transition: if either component is broken, security holds. Covers why hybrid approaches are mandated (or recommended) by ANSSI and NIST SP 800-227, six X.509 certificate format approaches (pure PQC ML-DSA, pure PQC SLH-DSA, composite dual-algorithm, alt-sig/catalyst, related certificates RFC 9763, and chameleon certificates), and hybrid KEM construction (X25519MLKEM768). Details the fundamental dilemma that PQC algorithms are newer and less battle-tested than classical ones, yet HNDL threats make waiting dangerous. Covers NIST SP 800-227 §4 implementation requirements: implicit rejection, constant-time decapsulation, DRBG quality, side-channel resistance, and domain-separated hybrid combiners. Out of scope: Pure PQC architectures, standalone hash-based signature deployment without hybrid partners, and algorithm replacements for non-KEM key agreement.

**Sub-topics keywords:** Hybrid cryptography, composite signatures, dual-algorithm certificates, X25519MLKEM768, RFC 9794 hybrid terminology, hybrid KEM combiner, ANSSI hybrid mandate, NIST SP 800-227, HKDF HMAC-based extractor, KMAC conditional, domain separation, implicit rejection, constant-time decapsulation, DRBG quality, side-channel resistance, certificate chain size, backward compatibility, related certificates RFC 9763, chameleon certificates, alt-sig catalyst, SubjectAltPublicKeyInfo extension.

## crypto-agility — Crypto Agility

**Scope.** This module teaches designing crypto-agile systems that can rapidly swap cryptographic algorithms, protocols, and implementations without code or infrastructure rewrites. Covers three dimensions of agility (algorithm, protocol, implementation), three architecture patterns (provider model, service mesh, external KMS/HSM), the CBOM (Cryptographic Bill of Materials) concept for discovering all crypto assets, and NIST's recommended 7-phase migration framework (Assessment, Risk Prioritization, Preparation, Testing, Hybrid Migration, Production, Monitoring). Includes real-world case studies from Cloudflare (~4% TLS handshake time increase with hybrid), Google Chrome (X25519MLKEM768 default since Chrome 131), and Apple iMessage (P-256 + Kyber-1024 initial keys, now standardized as ML-KEM-1024/768 per FIPS 203). Out of scope: Protocol-specific implementation details, cryptanalysis of individual algorithms, and hardware-only crypto agility approaches.

**Sub-topics keywords:** Crypto agility, algorithm agility, protocol agility, implementation agility, provider model, service mesh, external KMS HSM, CBOM cryptographic bill of materials, CycloneDX, 7-phase migration framework, assessment inventory, risk prioritization, hybrid migration, production deployment, monitoring, abstraction layer, crypto-agile APIs, JCA OpenSSL providers, configuration-driven algorithm selection, performance overhead, latency budget.

## crypto-mgmt-modernization — Cryptographic Management Modernization

**Scope.** This module teaches the management discipline of Cryptographic Posture Management (CPM) — a continuous, iterative program covering certificates, cryptographic libraries, application software, and key material. Distinct from crypto-agility (a technical capability: "can we change?") and from a Center of Excellence (operating model). CPM answers: "do we know what we have, is it healthy, can we prove it?" Covers five pillars of CPM (Inventory, Governance, Lifecycle/CLM, Observability, Assurance/FIPS), dual-loop iterative process, and ROI justification whether quantum arrives on schedule or never. Addresses three forcing functions: 47-day TLS certificate cadence by 2029 (CA/B Forum), FIPS 140-3 Level 3 self-test drift, and library EoL/CVE cadence (OpenSSL 1.1.1 EoL, OMB M-23-02 mandate). Out of scope: Specific crypto algorithms, detailed cryptanalysis, and FIPS module validation mechanics beyond the CPM governance context.

**Sub-topics keywords:** Cryptographic posture management CPM, CBOM cryptographic bill of materials, certificate lifecycle management CLM, ACME EST enrollment, certificate rotation, 47-day cadence, FIPS 140-3 Level 3, CMVP CAVP validation, library EoL end-of-life, CVE cryptographic vulnerabilities, inventory governance, continuous monitoring, observability metrics, shadow certificate discovery CT logs, root CA rotation, revocation propagation, OMB M-23-02, CNSA 2.0 deadlines.

## slh-dsa — SLH-DSA: Stateless Hash Signatures

**Scope.** Master FIPS 205 SLH-DSA: stateless hash-based digital signatures with no per-key state tracking (unlike stateful LMS/XMSS). Covers WOTS+ one-time signatures, FORS few-time signatures, hypertree architecture, twelve parameter sets from FIPS 205 Table 2 at NIST Levels 1/3/5 with SHA-2 and SHAKE variants, context strings for domain separation, deterministic signing mode, and HashSLH-DSA pre-hash variant. Covers the s vs f tradeoff (-s small variants produce smaller signatures but require more hash rounds per layer; -f fast variants use more layers for faster signing at larger signature cost). Signature sizes range from 7,856 bytes (SHA2-128s) to 49,856 bytes (SHAKE-256f). Security rests entirely on hash function properties (SHA-256, SHA-512, SHAKE256), not algebraic hardness. Covers CNSA 2.0 permitting SLH-DSA as stateless alternative to LMS/XMSS for firmware signing. Out of scope: WOTS chain-chain details, FORS randomness engineering, and hypertree optimization techniques.

**Sub-topics keywords:** SLH-DSA stateless hash-based signatures, FIPS 205, WOTS plus one-time signatures, FORS few-time signatures, hypertree, Merkle tree, parameter sets, SHA-2 SHAKE, security level, signature size, key size, signing speed, deterministic signing, context strings, domain separation, hash function collision resistance, CNSA 2.0, conservative security, stateless architecture, deployment simplicity.

## stateful-signatures — Stateful Hash Signatures

**Scope.** LMS (Leighton-Micali Signature) and XMSS (eXtended Merkle Signature Scheme) — most conservative post-quantum signature schemes standardized in NIST SP 800-208, deriving security solely from hash function properties without new cryptographic hardness assumptions. Covers Merkle tree structure (each leaf contains hash of unique one-time public key, tree root is overall public key, signatures include OTS signature plus authentication path), Winternitz OTS with parameter W controlling size/speed tradeoff (W=1 fastest/largest, W=8 slowest/smallest), and the catastrophic OTS reuse vulnerability (signing two different messages with same leaf enables private key recovery). Covers LMS/HSS (RFC 8554) with tree heights 5–25 (32 to 33M signatures), 56-byte public keys, 1.3–9.3 KB signature sizes, and XMSS/XMSS^MT with bitmask-based tree hash, tighter security proofs, and forward security. State management requirements (atomic updates, non-volatile storage, HSM monotonic counters). Comparison with SLH-DSA: LMS/XMSS offer smaller signatures and faster signing but require state tracking and bounded signature counts; SLH-DSA is stateless with unlimited signatures but larger/slower. Out of scope: Detailed Merkle tree construction, OTS key derivation mechanics, and HSS tree level coordination.

**Sub-topics keywords:** LMS Leighton-Micali Signature, HSS hierarchical signature scheme, XMSS eXtended Merkle signature scheme, XMSS^MT, NIST SP 800-208, Merkle tree, one-time signatures OTS, Winternitz OTS, state management, monotonic counter, signature size, key size, signing speed, forward security, RFC 8554 RFC 8391, bounded signatures, tree height, leaf index advancement, state loss risk, stateful vs stateless comparison.

## merkle-tree-certs — Merkle Tree Certificates

**Scope.** Addresses certificate bloat problem created by PQC signatures: ML-DSA-44 is 2,420 bytes vs ECDSA P-256 64 bytes (37× increase), causing 18–36 KB TLS chain overhead. Covers Merkle Tree Certificates (MTCs) solving this through batch signing (SHA-256 Merkle tree leaves = certificate assertions, only root signed with CA's PQC key, each cert holder receives compact inclusion proof ~736 bytes). Discusses MTC architecture: five roles (CA maintains issuance log + builds tree + signs checkpoints, Cosigner signs log view, Authenticating Party presents cert + proof, Relying Party verifies, Monitor watches log). Covers two certificate types: Standalone (inclusion proof + cosignatures, works without pre-sync) and Landmark (signatureless, requires pre-synced trusted subtrees, ~936 bytes vs 12,034 bytes traditional 92% reduction). Covers tradeoffs: Landmark requires periodic pre-sync (not suitable for offline devices), revocation via Revocation by Index, IETF standardization (draft-davidben-tls-merkle-tree-certs adopted by PLANTS WG). Connection to hash-based signatures: SLH-DSA/LMS/XMSS use Merkle trees internally; MTCs apply same principle at infrastructure layer. Out of scope: Tree rebalancing algorithms, cosigner threshold logic, and gossip protocol details for log consistency.

**Sub-topics keywords:** Merkle tree certificates, certificate bloat, PQC signature size, batch signing, inclusion proof, sibling hash chain, hash tree recomputation, domain separation, Merkle tree CA, cosigner attestation, append-only log, standlone certificates, landmark certificates, signatureless proof, revocation by index, draft-ietf-plants-merkle-tree-certs, IETF Hackathon test vectors, size reduction ratio, handshake overhead.

Out of scope: Quantum mechanics derivation, entanglement-based protocols beyond Micius examples, and photon detection engineering.

## api-security-jwt — API Security & JWT

**Scope.** Migration of JSON Web Token (JWT) infrastructure to post-quantum cryptography covering JWTs as foundation of modern API authentication (OAuth 2.0 access tokens, OpenID Connect ID tokens, DPoP proofs). Covers quantum vulnerability: RS256 (RSA), ES256 (ECDSA P-256), and EdDSA all broken by Shor's algorithm; ECDH-ES key agreement equally vulnerable; HMAC-based algorithms (HS256) remain quantum-safe but require shared secrets. Covers HNDL for JWTs: attackers capture signed JWTs today and forge signatures once quantum computers arrive (especially concerning for long-lived tokens: refresh tokens, long-expiry ID tokens). Covers PQC JWT signing with ML-DSA (IETF draft-ietf-jose-pqc): ML-DSA-44 (NIST Level 2, 2,420 B), ML-DSA-65 (Level 3, 3,309 B recommended), ML-DSA-87 (Level 5, 4,627 B). Covers PQC JWT key agreement with ML-KEM (replaces ECDH-ES in JWE), KEM encapsulation producing ciphertext + shared secret, HKDF derives Content Encryption Key.
